Most orgs have some version of this problem. A lead comes in from a web form. It lands in the default queue. The rep for that territory doesn't see it for two days. Or it gets assigned to someone who left six months ago because the assignment rule was configured in 2021 and nobody updated it.

The fix is straightforward: a record-triggered Flow that checks the lead's region on creation, assigns the owner to the right queue, and posts a Chatter message so the rep actually notices. Textbook admin work. And in Flow Builder, it's 45 minutes of clicking — minimum.

Claude Code does it in five minutes. Here's the exact workflow.

## The painful version

Here's what building this manually looks like in Flow Builder:

1. New Flow → Record-Triggered Flow → Object: Lead → Trigger: A record is created.
2. Add a Decision element. Three outcomes: North, South, West, plus Default. For each outcome, write the condition: `Region__c Equals [picklist value]`. Three conditions, typed by hand.
3. For each outcome, add an Assignment element to set `OwnerId`. You need the queue ID — not the name, the 18-character ID. Open a new browser tab. Setup → Queues. Copy each ID. Paste into each Assignment. Three tabs, three copies.
4. Realize that if a queue gets renamed or replaced, you'll have hardcoded IDs in the Flow XML that silently stop working. Decide to use a Get Records lookup instead. Redo step 3 with a Get Records element per outcome.
5. Add an Update Records element after each Assignment path to write `OwnerId` to the record. Wire each decision branch through it.
6. Add the Chatter Post. Action element. Post to a user. The recipient is `OwnerId` — but the Flow variable is the result of the Get Records lookup, and the mapping isn't obvious the first time you set it up.
7. Activate. Test with a lead in your sandbox. The "West" path fires even when Region is blank because you used `equals` instead of `equals (exact match)` — or you forgot to check null handling in the Default path.
8. Fix the logic. Retest. Activate again. Forty minutes have passed.

That's three regions. Add a fourth territory, a round-robin override, or a bypass for VIP accounts, and you're looking at another hour — easy.

## The Claude Code version

Open your terminal in your Salesforce DX project. Both orgs authenticated. One prompt:

```
Create a record-triggered Flow on Lead.

Trigger: when a Lead record is created.

Logic: check the Region__c picklist field.
  - If North, assign to the North Leads queue.
  - If South, assign to the South Leads queue.
  - If West, assign to the West Leads queue.
  - Default (blank or unrecognized): assign to the General Leads queue.

After assigning the owner, post a Chatter message to the new owner:
"New lead assigned to you: {Lead.FirstName} {Lead.LastName}
from {Lead.Company}. Check it out."

Name the Flow: Lead_Region_Assignment
Deploy to dev-org.
```

Claude Code, with the Salesforce DX MCP server wired in (see [the MCP setup post](/blog/connect-claude-code-to-salesforce-with-mcp)), runs four steps:

1. **Read the schema.** Queries `Region__c` from your Lead object. Pulls the actual picklist values. If your values are "North America", "EMEA", "APAC" instead of "North", "South", "West" — that's what ends up in the Flow conditions.
2. **Look up the queues.** Runs a SOQL query for the four queue names. Gets their record IDs. Generates the Flow with Get Records lookups, not hardcoded IDs — the Flow survives queue renames and org-to-org deploys.
3. **Generate the Flow XML.** Writes `Lead_Region_Assignment.flow-meta.xml` under `force-app/main/default/flows/`. Decision element, four Assignment elements, four Update Records elements, four Chatter Post actions. Shows you the diff.
4. **Deploy.** On your approval, runs `sf project deploy start --metadata Flow:Lead_Region_Assignment --target-org dev-org`. Prints the deploy ID and result.

Total: under five minutes. Most of that is you reading the diff.

## When NOT to use Claude Code for this

A few cases where a different tool wins:

- **You already have Salesforce's native Lead Assignment Rules set up and they work.** Assignment Rules are simple, zero-code, and native. If your logic is "assign by zip code range" or "round-robin across five reps with load balancing," Assignment Rules handle that without a Flow, without a prompt, and without a deploy. Don't build a Flow where a Setup screen already does the job.
- **You need true round-robin with load tracking.** A Flow can't maintain a rotating counter across concurrent saves without a shared custom field — and that field becomes a contention point under load. If you need real round-robin, look at a routing app (LeanData, Chili Piper, Revenue Grid) or an Apex trigger that handles it properly.
- **The assignment logic lives inside a managed package.** If your org uses a CPQ or sales engagement tool that controls Lead routing, fight the package configuration, not a competing Flow.
- **Your dev sandbox has empty queues.** Claude deploys a working Flow, but if the queue membership doesn't reflect reality, you can't validate the assignment end-to-end. Make sure queues exist with real or representative members before testing.

## Common pitfalls

Sharp edges that will bite you:

- **The Flow re-fires on every save, not just creation.** If you later change the trigger to "a record is created or updated," the assignment logic runs every time a rep touches the record. Add an entry condition: `{!$Record.IsNew} = true` to restrict it to new records only.
- **Region__c is blank on web-to-lead submissions.** Many web forms don't collect a region field. Leads arrive with a blank `Region__c` and fall to the Default path every time — your queues are useless. Map region from `State` or `PostalCode` at creation, or add a Chatter alert to a routing admin when the field is blank.
- **The Chatter post uses fields that can be null.** `Lead.Company` is not required in Salesforce. If it's blank, your post reads "New lead from ." Ask Claude to wrap it: `{!IF(ISBLANK(Lead.Company), 'an unknown company', Lead.Company)}` in the Chatter message template.
- **Two record-triggered Flows run on the same object.** If you already have a Flow firing on Lead creation — for, say, field defaulting or a welcome email — you now have two. Salesforce runs both; they don't block each other. But if both try to write `OwnerId`, the last one wins. Audit your existing Flows before adding another.
- **Deploy succeeds but queues don't receive email notifications.** Chatter notifications are separate from Salesforce email alerts. If your reps rely on email, you'll want a separate Email Alert element — or a Send Email action in the same Flow. The Chatter post is visible in the record feed but doesn't send push or email by default.
