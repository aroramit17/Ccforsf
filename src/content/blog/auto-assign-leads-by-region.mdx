Every Salesforce org eventually gets the same question from sales leadership: "Why did that lead go to the wrong person again?" The answer is usually that someone covered for a rep on vacation, the Lead Assignment Rule hasn't been touched since 2021, and nobody's sure which queue actually belongs to which region anymore.

The fix is a record-triggered Flow: when a Lead comes in, check the state, route to the right queue, post to a team Chatter group so the handoff is visible. It's maybe two hours of work in Setup and Flow Builder. Or it's one prompt and five minutes.

## The painful version

Here's what building this automation looks like without Claude Code:

1. Go to Setup → Lead Assignment Rules → New. Name the rule. Save.
2. Click into the rule to add entries. Entry #1: West Region. Add state criteria — California, Oregon, Washington, Nevada, Arizona, Utah, Idaho, Montana, Wyoming, Colorado. Select the queue. Hope the dropdown label matches what your ops team calls it.
3. Repeat for Central. Fourteen states. Enter them one at a time. Save.
4. Repeat for East (Default). Save.
5. Realize Lead Assignment Rules have no Chatter notification element. You need a separate Flow for that.
6. Open Flow Builder. New Flow. Record-Triggered. Lead object. Trigger on create or update.
7. Set entry criteria so this doesn't fire on every single Lead update for the rest of time.
8. Build a Decision element. Three outcomes. One per region.
9. Connect Assignment elements in each branch to set `OwnerId` to the queue ID. Look up the queue IDs — they're GUIDs, not names.
10. Add Post to Chatter actions in each branch. Type the message text. Use merge fields correctly.
11. Add fault paths on every DML element. Both of them. If you forget one, the Flow fails silently.
12. Activate. Create a test Lead with a California address. The Flow fires but the Chatter post fails — the group ID is wrong. The West Chatter group is named `Sales_West_US`, not `West_Sales_Team`.
13. Fix it. Re-test. Done. Ninety minutes have passed.

The part that's exhausting isn't any single step. It's the context-switching: Setup tab, Flow Builder tab, back to Setup to look up a queue ID, back to Flow Builder to paste it.

## The Claude Code version

Open your terminal. Your Salesforce DX project is connected to your org via the [Salesforce DX MCP server](/blog/connect-claude-code-to-salesforce-with-mcp). Prompt:

```
Build a record-triggered Flow on the Lead object. Trigger on create or update.
Entry criteria: IsNew = true OR ISCHANGED(State).

When a Lead is created or its State changes:
- West Region (CA, OR, WA, NV, AZ, UT, ID, MT, WY, CO): assign to queue "Sales_West_AMER",
  post to Chatter group "West_Sales_Team" — message: "New lead {Lead.Name} from {Lead.State} — routed to West queue."
- Central Region (TX, OK, KS, NE, SD, ND, MN, IA, MO, IL, WI, MI, IN, OH): assign to "Sales_Central_AMER",
  post to "Central_Sales_Team".
- Everything else: assign to "Sales_East_AMER", post to "East_Sales_Team".

Add fault paths on all DML and action elements.
Deploy to dev-org.
```

Claude runs four steps:

1. **Look up real IDs.** Queries `Group WHERE Type = 'Queue'` and `CollaborationGroup` in your org. Resolves queue names to Salesforce IDs before writing a single line of metadata. If a name doesn't match, it tells you before generating anything.
2. **Generate the Flow XML.** Creates `force-app/main/default/flows/Lead_Region_Assignment.flow-meta.xml`. Decision element with three outcomes, Assignment elements, Post to Chatter actions, fault paths on every DML element.
3. **Diff review.** Reads back the generated XML and confirms the state lists, queue IDs, and Chatter group IDs match what it found. Reports any discrepancy.
4. **Deploy.** Runs `sf project deploy start --metadata Flow:Lead_Region_Assignment --target-org dev-org`. Outputs the deploy ID and confirms the Flow is active.

Total time: 3–5 minutes, most of it Salesforce CLI network latency.

## The part people forget: fault paths

In nine out of ten hand-built Flows, at least one DML or action element has no fault path connected. The Flow fails at runtime, Salesforce swallows the error, and a Lead gets created with no owner assignment and no Chatter notification. Nobody notices until sales asks why three leads sat in the default queue for a week.

Claude connects fault paths on every element that can fail — Assignment, Post to Chatter, any Get Records. If you want the fault path to do something beyond default behavior (send an email to an admin, write to an error log object, post to a monitoring Chatter group), add it to the prompt. Claude can add a Post to Platform Event element on each fault path that feeds into a monitoring Flow.

This is the kind of thing you know you should do when building by hand. You also know you're going to skip it because you've been building this Flow for an hour and you just want it done.

## When NOT to use Claude Code for lead assignment

A few cases where another tool wins:

- **You have an existing assignment rule with 50+ criteria entries.** If it already works, don't touch it. Claude can help you migrate it to a Flow eventually, but that's a project, not a quick build.
- **Your marketing automation sets owner on create.** If HubSpot, Marketo, or Pardot is already setting `OwnerId` when it creates the Lead, a record-triggered Flow that also sets `OwnerId` will fire on the same record and overwrite the MA assignment. Map the ownership model first. One system owns the assignment.
- **You need round-robin within a region.** Native Flow has no round-robin element. You need Apex. Claude can scaffold the Apex invocable action, but that's a 30-minute build, not a 5-minute one. Do it separately once the region routing is stable.
- **Your org is on Salesforce Government Cloud with strict metadata approval gates.** The generation benefit is the same, but the deploy path goes through your change advisory board as usual. Generate with Claude, review through your normal process.

## Common pitfalls

- **No entry criteria.** A record-triggered Flow without entry criteria fires on every Lead save — including bulk data imports of 50,000 records. Always scope with `IsNew = true` or `ISCHANGED(State)` as shown above.
- **Queue assignment vs. owner display.** When you assign a Lead to a queue, `OwnerId` becomes the queue's group ID. If the Chatter post reads `{!$Record.OwnerId}` immediately after the assignment, it shows an ID string, not a name. Use `{!$Record.Owner.Name}` instead, or query the queue's `Name` via a Get Records element before posting.
- **Scope creep on the trigger condition.** "Trigger on create or update" without `ISCHANGED(State)` means every field edit fires the Flow. If a rep updates the Lead description six months later, the lead re-routes. Narrow it.
- **Missing the default branch test.** West and Central get tested. East (the default) gets skipped. Test all three branches. Prompt Claude to generate test records covering each one.
- **Deploying Active to prod without a sandbox run.** The Flow deploys as Active. If the Chatter group IDs in prod don't match the ones in dev, the fault paths fire silently on every lead creation. Test in sandbox against real leads before promoting.
